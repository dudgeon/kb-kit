Decision 1 — voice

Same two passages, three voices. Pick the one you'd want the whole page written in.

Direction A

Technical README

Terse and factual. States what things are; links do the explaining.

opening kb-kit is a template repository for running a knowledge base as markdown in git. It contains an OKF v0.1 bundle (kb/), four agent skills (setup, ingest, query, lint), a kb-card.md describing the KB, and a static site that renders the bundle client-side. Fork it, run the setup skill, start ingesting.
composition What's in the repo kb/ — the bundle: frontmatter-typed pages, _index.md catalog, _log.md history, inbox queue, raw archive. skills/ — the four workflows, agent-agnostic. kb-card.md — declared scope + lint-computed health. Site: two HTML files, no build framework; Pages serves the branch.
Direction B

Field guide

Educational narrator. Explains the concept before the feature; teaches as it goes.

opening A knowledge base your team and its agents maintain together, as plain markdown in a git repo. The idea (from Karpathy's LLM-wiki pattern): instead of retrieval at question-time, an agent continuously files what your team learns into a small wiki it keeps consistent — you read it, the agent writes it, git reviews it. kb-kit is that idea packaged: fork it and the structure, the workflows, and the browsing site are already in place.
composition What you get, and why each piece exists A bundle (kb/) in Google's Open Knowledge Format, because a KB should be readable by any tool that can read files. Four skills — setup, ingest, query, lint — because a KB is an operating loop, not a folder. A kb-card, because in a company of many KBs, each one needs to say what it is before you read it. And a site, because casual readers shouldn't need a clone.
Direction C

Systems doc

Architecture-first. Leads with layers, contracts, and lineage; reads like a design doc.

opening kb-kit implements a three-layer context repo. Layer 1: immutable raw sources (kb/sources/raw/, fed by an inbox queue and GitHub issues). Layer 2: the distilled wiki (kb/, typed frontmatter, relative links only). Layer 3: the schema (AGENTS.md) that tells any agent how to operate the loop — ingest → query → lint. A static viewer and a kb-card expose the repo to humans and crawlers.
composition Contracts Format: OKF v0.1, two documented deviations (relative links; _index.md/_log.md). State-by-location: inbox = unprocessed, raw = preserved. Deployment: branch-served Pages, .nojekyll load-bearing. All automation is deterministic scripts; agentic work runs in the owner's own session. Lineage: Karpathy llm-wiki → Google OKF → agents.md → model cards.
